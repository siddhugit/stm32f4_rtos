Task states

Blocking delay APIS (Doesn't hog CPU)
vTaskDelay() - This creates a delay relative to the time when it gets called
vTaskDelayUntil() - This creates a delay at an absolute time specified as arguments. Use this to run tasks periodically.
Both the delay APIs take tick_count and not time or duration_elapsed
pdMS_TO_TICKS is the macro to convert to tick_count from duration_elapsed in millisecons
xTaskGetTickCount() - Function to get current number of ticks
duration_elapsed(in seconds) =  number_of_ticks / configTICK_RATE_HZ
duration_elapsed(in milliseconds) =  (number_of_ticks / configTICK_RATE_HZ)*1000

Hardware priorities are priorities given to the interrupts and exceptions
Exceptions are generated by hardware such as systick,pendsv interrupts or hard fault
Interrupts are generally associatied with peripherals like uart, spi etc
In Arm cortex Mx processors, priority is the niceness value, i.e. higher the value lower the priority
In FreeRTOS, Task priority is higher if value is higher

configKERNEL_INTERRUPT_PRIORITY - The hardware interrupt priority used by FreeRTOS for systick, pendsv ISRs
configMAX_SYSCALL_INTERRUPT_PRIORITY -  The maximum priority an interrupt can have whose ISR calls "FromISR" versions of FreeRTOS APIs

Higher priority interrupt pre-empts currently running lower interrupt

Two modes of processor
(1) Thread mode
(2) Handler mode

Why "FromISR" versions of APIs exists?
(1) Because there should not blocking functions calls from ISRs 

FreeRTOS hook functions
(1) Idle task hook fucntion - callback which is called for each cycle of idle task ( void vApplicationIdleHook() )
(2) RTOS tick hook function - Called every time rtos tick interrupt occurs i.e. systick interrupt (void vApplicationTickHook())
(3) malloc failed hook function - void vApplicationMallocFailedHook( void )
(4) stack overflow hook function

 Timers
 Hardware timers are those which uses timer peripherals like TIM1,Tim2..TIM6 etc
 hardware must be configured e.g. reload values, interrupt handlers etc in order to use hardware timers
 A high resolution can be achieved when using hardware timers e.g. microseconds or even nanoseconds
 
 A software timers is software based and it is bounded by RTOS_TICK_RATE_HZ ( e.g. 1 millisecond )
 Create timer
 TimerHandle_t xTimerCreate
                 ( const char * const pcTimerName,
                   const TickType_t xTimerPeriod,
                   const UBaseType_t uxAutoReload,
                   void * const pvTimerID,
                   TimerCallbackFunction_t pxCallbackFunction );
uxAutoReload - set false for single shot or true for continous timer
timer id is used in case same call back is used for multiple timers
pvTimerGetTimerID() is used to get timer id

There is a queue maintained by timer service task to manage all the timers

xTimerStart() and xTimerStop() uses the timer queue in the implementation

 BaseType_t xTimerStart( TimerHandle_t xTimer,
                            TickType_t xBlockTime );
xBlockTime - This corresponds to the waiting time for sending the start command to the timer queue
BaseType_t xTimerStop( TimerHandle_t xTimer,
                        TickType_t xBlockTime );
xBlockTime - This corresponds to the waiting time for sending the stop command to the timer queue
