FreeRTOS Notes

Modify these resources for stm32 so that they can used by FreeRTOS

Systick - Assign a different time source to stm32 e.g. TIM6 (SYS->Timebase source)
Disable interrupt handlers code generation for SVC, PendSV and Systick (NVIC)
4 bits for priority and 0 for sub-priority ( NVIC->Priority Group)
Call vInitGroupPriorityValue() from HAL_MaspInit()

Task states
RUNNING,READY,BLOCKED,SUSPENDED

Blocking delay APIS (Doesn't hog CPU)
vTaskDelay() - This creates a delay relative to the time when it gets called
vTaskDelayUntil() - This creates a delay at an absolute time specified as arguments. Use this to run tasks periodically.
Both the delay APIs take tick_count and not time or duration_elapsed
pdMS_TO_TICKS is the macro to convert to tick_count from duration_elapsed in millisecons
xTaskGetTickCount() - Function to get current number of ticks
duration_elapsed(in seconds) =  number_of_ticks / configTICK_RATE_HZ
duration_elapsed(in milliseconds) =  (number_of_ticks / configTICK_RATE_HZ)*1000

Task wait and notify APIs
xTaskNotifyWait()
xTaskNotify()

Each of the heap_*.c (1-5) implements a different heap management algorithms
heap_5.c is mostly used.

FreeRTOS mutual exclusion and synchronization services
Semaphores (Counting and binary)
Queues
Mutex

vSuspendAll() - disables context switching
vResumeAll() - enables context switching

Hardware priorities are priorities given to the interrupts and exceptions
Exceptions are generated by hardware such as systick,pendsv interrupts or hard fault
Interrupts are generally associatied with peripherals like uart, spi etc
In Arm cortex Mx processors, priority is the niceness value, i.e. higher the value lower the priority
In FreeRTOS, Task priority is higher if value is higher

configKERNEL_INTERRUPT_PRIORITY - The hardware interrupt priority used by FreeRTOS for systick, pendsv ISRs
configMAX_SYSCALL_INTERRUPT_PRIORITY -  The maximum priority an interrupt can have whose ISR calls "FromISR" versions of FreeRTOS APIs

Higher priority interrupt pre-empts currently running lower interrupt

Two modes of processor
(1) Thread mode
(2) Handler mode

Why "FromISR" versions of APIs exists?
(1) Because there should not blocking functions calls from ISRs
